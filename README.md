![AI Friendly](https://img.shields.io/badge/AI-Friendly-purple)

一个面向青龙脚本的通用 WxPusher 推送框架。

它提供统一的 HTML 面板渲染器、明暗主题支持、可复用调用模板，以及面向 AI 的标准化接入规范，适合多账号挂机、签到、积分查询等脚本场景。

## Features

- 统一的 WxPusher HTML 推送入口
- 现代化面板设计，支持 `light` / `dark` / `auto`
- 面向多账号脚本的简洁信息流结构
- 支持汇总指标、账号卡片、任务明细和扩展字段
- 提供 AI 可直接遵循的接入规范
- 提供最小调用模板，便于快速接入新脚本
- 适配青龙面板部署和子目录脚本结构
- 内置内容清洗、长度控制和发送结果校验

## Files

- `wxpusher_utils.py`: 核心推送与 HTML 面板渲染模块
- `WXPUSHER_TEMPLATE.py`: 最小可复用调用模板
- `WXPUSHER_AI_GUIDE.md`: 面向 AI 和开发者的统一接入规范
- `README.md`: 用户使用说明
- `LICENSE`: MIT 开源协议

## Quick Start

### 1. 准备文件

将以下文件放入你的青龙脚本目录中：

- `wxpusher_utils.py`
- `WXPUSHER_TEMPLATE.py`
- `WXPUSHER_AI_GUIDE.md`（可选，供 AI 或开发者参考）

### 2. 配置环境变量

在青龙面板中添加以下环境变量：

- `WXPUSHER_APP_TOKEN`
- `WXPUSHER_UID`

说明：

- `WXPUSHER_APP_TOKEN`：WxPusher 应用的 `appToken`
- `WXPUSHER_UID`：目标接收用户的 UID
- `WXPUSHER_UID` 支持单个 UID，也支持多个 UID 以英文逗号分隔

示例：

```env
WXPUSHER_APP_TOKEN=AT_xxxxxxxxxxxxxxxxx
WXPUSHER_UID=UID_xxxxxxxxxxxxxxxxx
```

多个 UID 示例：

```env
WXPUSHER_UID=UID_xxx1,UID_xxx2,UID_xxx3
```

### 3. 编写业务脚本

业务脚本只负责产出结构化数据，再调用公共入口发送：

```python
from wxpusher_utils import PanelItem, PanelMetric, PanelTag, push_html_panel

metrics = [
    PanelMetric("总账号", "3", "primary"),
    PanelMetric("成功", "2", "success"),
    PanelMetric("失败", "1", "danger"),
]

items = [
    PanelItem(
        title="账号A",
        value="+12",
        desc="今日任务完成",
        meta="当前积分 328",
        status="success",
        tags=[PanelTag("签到成功", "success")],
    ),
    PanelItem(
        title="账号B",
        value="Cookie失效",
        desc="本次任务已跳过",
        meta="需要重新获取凭证",
        status="danger",
        tags=[PanelTag("需要登录", "danger")],
    ),
]

push_html_panel(
    summary="今日执行完成：2 正常 / 1 异常",
    title="脚本执行报告",
    subtitle="QingLong Task",
    metrics=metrics,
    items=items,
    theme="auto",
    footer="Generated by QingLong WxPusher Panel",
)
```

### 4. 在青龙中运行

直接在青龙面板创建任务并执行脚本即可。

如果脚本与配置文件都放在青龙脚本根目录下，直接运行即可。

## QingLong Deployment

### 推荐目录结构

如果你只是简单使用，推荐结构如下：

```text
/ql/data/scripts/
├── wxpusher_utils.py
├── WXPUSHER_TEMPLATE.py
├── your_script.py
```

这种结构下，业务脚本可以直接：

```python
from wxpusher_utils import push_html_panel
```

### 子目录部署

如果某些脚本的账号信息较长，不方便放在环境变量中管理，你可以单独建立文件夹来存放脚本和配置文件，例如：

```text
/ql/data/scripts/
├── wxpusher_utils.py
├── project_a/
│   ├── run.py
│   ├── accounts.json
│   └── config.json
├── project_b/
│   ├── main.py
│   └── tokens.txt
```

此时，子目录中的脚本需要在开头动态追加模块搜索路径，以便调用根目录下的 `wxpusher_utils.py`。

推荐写法：

```python
import os
import sys

CURRENT_DIR = os.path.dirname(os.path.abspath(__file__))
SCRIPTS_ROOT = os.path.abspath(os.path.join(CURRENT_DIR, ".."))

if SCRIPTS_ROOT not in sys.path:
    sys.path.append(SCRIPTS_ROOT)

from wxpusher_utils import push_html_panel
```

### 更深层目录

如果脚本位于二级、三级甚至更深目录中，只需要按实际层级调整：

```python
SCRIPTS_ROOT = os.path.abspath(os.path.join(CURRENT_DIR, "..", ".."))
```

核心原则：

- 公共模块 `wxpusher_utils.py` 放在青龙 `scripts` 根目录
- 业务脚本可位于任意子目录
- 业务脚本通过动态追加 `sys.path` 调用公共模块

## Theme Modes

当前渲染器支持 3 种主题模式：

- `light`: 浅色主题
- `dark`: 深色主题
- `auto`: 自动模式

推荐正式脚本统一使用：

```python
theme="auto"
```

这样在不同客户端环境下能获得更自然的展示效果。

## Data Model

当前推送模型由以下几个核心结构组成：

### PanelMetric

用于头部汇总指标，例如：

- 总账号数
- 成功数
- 失败数
- 待处理数

示例：

```python
PanelMetric("成功", "8", "success")
```

### PanelTag

用于短标签信息，例如：

- 签到成功
- Cookie 失效
- 待登录
- 余额更新

示例：

```python
PanelTag("签到成功", "success")
```

### PanelItem

用于单个账号或单个对象的主要展示卡片。

常见字段：

- `title`: 账号名或对象名
- `value`: 主结果
- `desc`: 简要说明
- `meta`: 次级补充信息
- `status`: 状态色
- `tags`: 标签列表
- `lines`: 补充文本
- `tasks`: 可选任务明细
- `extras`: 可选扩展字段

示例：

```python
PanelItem(
    title="账号A",
    value="+12",
    desc="今日任务完成",
    meta="当前积分 328",
    status="success",
    tags=[PanelTag("签到成功", "success")],
    lines=["已完成日常任务", "无异常"],
)
```

### 状态值建议

统一使用以下语义：

- `primary`
- `success`
- `warning`
- `danger`
- `neutral`

推荐映射：

- 成功：`success`
- 已完成但无变化：`neutral`
- 待处理或部分完成：`warning`
- 失败或异常：`danger`

## Design Principles

这个项目的设计目标不是单纯“把消息发出去”，而是解决青龙多账号脚本中更本质的问题：

- 让用户第一眼看懂执行结果
- 让异常账号更容易被定位
- 让后续新脚本无需重复设计推送逻辑
- 让 AI 或开发者可以按统一协议稳定接入

因此它遵循以下原则：

- 业务逻辑与展示逻辑分离
- 数据结构稳定，渲染器可演进
- 信息流简洁，不堆日志墙
- 主题可切换，风格统一
- 扩展字段有正式承载位，不靠字符串硬塞

## For AI Integration

如果你打算让 AI 帮你编写新的青龙脚本，建议直接让它遵循仓库中的：

- `WXPUSHER_AI_GUIDE.md`

这个文件定义了统一的接入标准，包括：

- 强制规则
- 数据结构协议
- 调用模板
- 验收要求
- 常见错误与禁止事项

推荐做法：

- 业务脚本只负责采集业务数据
- 结果统一映射到 `metrics + items`
- 正式推送统一调用 `push_html_panel()`
- 不再自行拼接 HTML
- 不再自行直接 `requests.post()` 到 WxPusher 接口

## Example Use Cases

这套框架适合以下场景：

- 多账号签到脚本
- 多账号挂机积分脚本
- 多账号任务执行报告
- 余额查询与资产变更提醒
- 抽奖、道具、等级等附加信息展示
- 需要统一风格和统一接入方式的青龙自动化脚本

## Notes

- 正式推送建议统一使用 HTML 面板
- 控制台日志建议保留纯文本输出
- 单账号异常不应中断整批执行
- 长文本内容应尽量在业务侧先做压缩和提炼
- 若后续需要更复杂的展示，可在现有数据结构基础上继续扩展渲染器
